    🔖 [Page 4 - Page 5]：[网络体系结构为何采用分层]

    • 🧠 核心术语 (Concepts)：
    • Network architecture（网络体系结构）：网络按层次栈组织。
    • Layering（分层）：把复杂协调问题拆成局部问题。
    • Encapsulation（封装）：下层向上层提供服务并隐藏实现细节。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：总体架构层面
    • 运行逻辑：1. 网络通信协调复杂 -> 2. 采用分层降低设计复杂度 -> 3. 每层在下层基础上构建并向上提供服务
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 为什么分层能提升互连互通？简答：分层把整体复杂性拆解，接口清晰，便于异构系统协同。
    • 封装的直接价值是什么？简答：上层只依赖服务，不依赖下层内部状态和算法。

    🔖 [Page 6 - Page 8]：[协议、对等实体、接口与协议栈]

    • 🧠 核心术语 (Concepts)：
    • Protocol（协议）：通信双方关于“如何通信”的约定。
    • Peers（对等实体）：不同主机对应层的实体（可为进程或硬件）。
    • Interface（接口）：下层向上层开放的原语与服务。
    • Protocol Stack（协议栈）：系统使用的协议列表。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：跨层通用
    • 运行逻辑：1. 对等层之间按协议“虚拟通信” -> 2. 相邻层通过接口调用服务 -> 3. 整体由层+协议构成体系结构
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 协议与接口边界在哪里？简答：协议约束对等实体消息规则；接口约束上下层服务访问方式。
    • 为什么“架构”不等于“实现”？简答：架构定义层与协议关系，不绑定具体实现细节。

    🔖 [Page 10 - Page 11]：[虚拟通信与信息流]

    • 🧠 核心术语 (Concepts)：
    • Virtual communication（虚拟通信）：除最底层外的逻辑对等通信。
    • Actual communication（实际通信）：物理层真实传输。
    • Information flow：支撑高层虚拟通信的信息流过程。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：层间总体机制
    • 运行逻辑：高层看见的是对等通信，真实数据逐层下传后在底层实际发送，再逐层上传恢复语义。
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 为什么高层“看起来直连”？简答：中间细节被分层与封装屏蔽。
    • 虚拟通信依赖什么前提？简答：依赖各层按协议正确处理并转交数据。

    🔖 [Page 12 - Page 14]：[封装细节：消息到段/包/帧]

    • 🧠 核心术语 (Concepts)：
    • Header（首部）：控制信息（如序号、长度、时间等）。
    • Packet：三层分片后单元。
    • Frame：二层加首尾部后的传输单元。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：应用层 / 传输层 / 网络层 / 链路层 / 物理层
    • 运行逻辑：1. 五层产生命令 M 给四层 -> 2. 四层加首部 -> 3. 三层因尺寸限制分包并加三层首部、选择外发链路 -> 4. 二层加首部与尾部 -> 5. 一层物理发送
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 三层为什么要“分包”？简答：因为三层协议对可传消息大小有限制。
    • 二层为什么要加 trailer？简答：课件给出“加首尾部”机制形成链路层传输单元。

    🔖 [Page 15]：[网络体系结构复盘]

    • 🧠 核心术语 (Concepts)：
    • 分层目标：简化设计与实现、便于互连互通。
    • U 形数据流：发送端自顶向下、接收端自底向上。
    • 封装控制信息：由协议定义。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：跨层总览
    • 运行逻辑：对等实体按协议通信，最底层实际通信，其余层虚拟通信，系统端到端形成 U 形信息流。
    • 🎯 考点与高频面试题 (Exam Focus)：
    • “最底层实际通信”如何理解？简答：真实比特传输只在物理介质发生。
    • 封装字段由谁定义？简答：由该层协议定义。

    🔖 [Page 16]：[分层设计核心问题]

    • 🧠 核心术语 (Concepts)：
    • Reliability：差错检测和恢复。
    • Routing：路径选择。
    • Addressing/Naming：标识发送方与接收方。
    • Internetworking：网络互通性。
    • Scalability：可扩展性。
    • Statistical multiplexing：按需分配资源。
    • Flow control：避免快发送方淹没慢接收方。
    • QoS：服务质量。
    • Security：安全性。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：跨层设计议题
    • 运行逻辑：分层用于隐藏实现并支持演进，同时处理资源分配与性能/安全目标。
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 流量控制为何必要？简答：快发送方可能淹没慢接收方。
    • 可扩展性为何关键？简答：网络规模增长要求机制可持续演进。

    🔖 [Page 17 - Page 19]：[面向连接与无连接服务分类]

    • 🧠 核心术语 (Concepts)：
    • Connection-oriented service：建立连接并可协商。
    • Reliable vs Unreliable：是否确认可靠。
    • Message sequence vs Byte stream：前者保留消息边界，后者无边界。
    • Store-and-forward vs Cut-through：两种交换方式。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：服务模型层面
    • 运行逻辑：按是否建连、是否可靠、边界保持与交换方式进行服务分类（课件给出六种类型）。
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 字节流为何“不保留边界”？简答：连接只呈现连续字节序列。
    • 交换方式差异影响什么？简答：影响转发处理方式与时延形态。

    🔖 [Page 20]：[服务、用户与 SAP]

    • 🧠 核心术语 (Concepts)：
    • Service provider：第 n 层。
    • Service user：第 n+1 层。
    • SAP（Service Access Point）：上层访问下层服务的位置。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：相邻层接口关系
    • 运行逻辑：1. n 层实现服务 -> 2. n+1 层通过 n 层 SAP 调用
    • 🎯 考点与高频面试题 (Exam Focus)：
    • SAP 的工程意义是什么？简答：提供明确访问点，隔离层间实现细节。
    • 为什么要区分 provider/user？简答：明确职责与调用方向。

    🔖 [Page 21 - Page 23]：[服务原语与呼叫流程]

    • 🧠 核心术语 (Concepts)：
    • Service primitives：服务规范的操作集合。
    • CONNECT.request / indication / response / confirm：连接建立阶段原语。
    • DATA.request / indication：数据传输阶段原语。
    • DISCONNECT.request / indication：连接释放阶段原语。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：接口与服务调用
    • 运行逻辑：1. CONNECT.request -> 2. CONNECT.indication -> 3. CONNECT.response -> 4. CONNECT.confirm -> 5. DATA 交换 -> 6. DISCONNECT
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 为什么 CONNECT 需要四个阶段原语？简答：要覆盖请求、通知、应答、确认四类状态变化。
    • DATA 与 CONNECT 的职责差异？简答：CONNECT 负责会话建立，DATA 负责载荷交换。

    🔖 [Page 24 - Page 25]：[简单面向连接服务实现视角]

    • 🧠 核心术语 (Concepts)：
    • System call：协议栈在 OS 内时，原语通常表现为系统调用（API）。
    • Process：程序的一次执行。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：操作系统与协议栈交互
    • 运行逻辑：服务原语通过系统调用对应用暴露，支撑简单 client-server 的建连、传输和交互。
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 为什么服务原语常映射为系统调用？简答：协议栈在 OS 中，需要 OS API 暴露能力给应用。
    • 面向连接交互最小要素有哪些？简答：建连、数据交换、断连。

    🔖 [Page 26 - Page 28]：[DNS 与 Web 访问中的“协议+服务”]

    • 🧠 核心术语 (Concepts)：
    • DNS：域名到 IP 的翻译机制。
    • UDP：课件示例中为 DNS 提供无连接服务。
    • TCP：课件示例中为 Web 请求/响应提供面向连接服务。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：应用层协议 + 传输层服务
    • 运行逻辑：1. 输入域名并经 DNS 获取 IP（示例：114.255.40.66） -> 2. 浏览器与 Web 服务器经 TCP 建连 -> 3. 请求主页并返回主页数据
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 同一业务流程里为何先 UDP 后 TCP？简答：课件示例中 DNS 由 UDP 承载、Web 由 TCP 承载。
    • 该例如何体现“服务由协议实现”？简答：应用协议分别依赖下层 UDP/TCP 服务完成通信。

    🔖 [Page 29 - Page 31]：[Service 与 Protocol 的解耦]

    • 🧠 核心术语 (Concepts)：
    • Service：层对用户可执行的操作定义，不描述实现细节。
    • Protocol：对等实体交换消息的格式与语义规则。
    • PDU（Protocol Data Unit）：对等实体间交换的信息单元。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：架构抽象层面
    • 运行逻辑：1. 下层通过接口向上提供服务 -> 2. 对等实体通过协议实现服务 -> 3. 服务不变时可替换协议
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 为什么说 service/protocol “completely decoupled”？简答：服务定义“做什么”，协议定义“怎么做”。
    • 协议升级的边界条件是什么？简答：上层可见服务语义不变。

    🔖 [Page 32]：[协议标准的三要素]

    • 🧠 核心术语 (Concepts)：
    • Syntax：协议消息格式。
    • Semantics：字段及取值含义。
    • Synchronization/Timing：事件发生顺序说明。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：协议规范层面
    • 运行逻辑：协议标准通过语法、语义、同步三维共同约束行为。
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 仅定义消息格式为什么不够？简答：还需定义字段含义与时序才能互操作。
    • 时序约束常影响什么？简答：影响双方状态推进与事件触发顺序。

    🔖 [Page 33]：[服务原语与 PDU 的电话类比]

    • 🧠 核心术语 (Concepts)：
    • Service primitives（Con/Data/Disc 系列）：用户与服务提供者交互的操作。
    • PDU：服务提供者之间交换的协议数据单元。
    • ⚙️ 协议与机制解析 (Mechanisms)：
    • 所属分层：服务调用与对等交换关系
    • 运行逻辑：1. 用户通过接口触发原语 -> 2. 服务提供者之间交换 PDU -> 3. 完成请求/通知/数据/断开
    • 🎯 考点与高频面试题 (Exam Focus)：
    • 原语与 PDU 的本质区别？简答：原语是“用户-本层服务”调用，PDU 是“对等实体-对等实体”交换。
    • 电话类比解决了什么理解难点？简答：区分接口调用流程与网络内部协议交换流程。
