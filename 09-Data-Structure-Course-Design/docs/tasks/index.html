    项目任务清单

    1. 使用说明

    本文件用于管理小组当前任务、负责人、优先级、状态和截止时间。

    状态说明：

    • todo：未开始
    • doing：进行中
    • done：已完成
    • blocked：被阻塞

    优先级说明：

    • P0：必须优先完成，直接影响项目推进
    • P1：重要，但可在 P0 后推进
    • P2：增强项或后置项

    2. 当前阶段

    当前阶段：项目启动与第一轮设计

    当前目标：

    • 明确项目最小范围（MVP）
    • 完成三人分工落地
    • 建立核心文档骨架
    • 确定算法方向和实现边界

    阶段完成标准：

    • docs/prd.md 有初稿
    • docs/schema.md / docs/graph-model.md 有初稿
    • docs/architecture.md / docs/api.md 有初稿
    • 代码目录结构已确定
    • 三人都清楚自己本周任务

    3. 角色对应

    • 组员 A：Lead / PM / 文档总控
    • 组员 B：Data & Algorithm
    • 组员 C：System & Implementation

    5. 第一阶段交付要求

    组员 A 本周必须交付

    • docs/tasks.md
    • docs/prd.md 初稿
    • docs/progress.md 初稿

    组员 B 本周必须交付

    • docs/schema.md 初稿
    • docs/graph-model.md 初稿
    • docs/algorithm-design.md 初稿

    组员 C 本周必须交付

    • docs/architecture.md 初稿
    • docs/api.md 初稿
    • src/ 基础目录结构

    6. 下阶段预告（暂不立即执行）

    当第一阶段通过后，进入“最小可运行闭环实现”阶段，优先做：

    1. 旅游推荐功能最小版本
    2. 路线规划功能最小版本
    3. 场所查询功能最小版本

    进入条件：

    • 需求范围不再频繁变动
    • 数据结构方案已确认
    • 接口边界已确认
    • 技术栈已确认

    7. 阻塞处理规则

    如果出现以下情况，任务状态改为 blocked，并在群里同步：

    • 需求不清，无法继续
    • 依赖文档未完成
    • 字段或接口发生冲突
    • 算法方案与实现方案不一致

    处理原则：

    • 组员 A 负责收敛问题并更新任务状态
    • 组员 B 负责解释算法相关阻塞
    • 组员 C 负责解释实现相关阻塞

    8. 更新规则

    • 任务状态至少每周更新一次。
    • 完成任务后，负责人需同步修改本文件。
    • 若新增任务，直接追加到任务表并分配负责人。
    • 若调整优先级，必须在群里说明原因。
