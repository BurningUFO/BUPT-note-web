    三人小组 AI 协作落地计划书

    1. 文档用途

    这份计划书用于指导 3 人小组完成《数据结构课程设计》中的“个性化旅游系统”项目。

    目标是：

    • 不搭建复杂的多 Agent 平台。
    • 使用“人类主导分工 + 各自 AI 辅助 + 共享 Git 仓库协作”的方式推进项目。
    • 让每位组员清楚自己的职责、交付物和协作方式。
    • 让项目过程可记录、可追踪、可答辩、可验收。

    这份计划书可以直接发给组员使用，作为项目执行规范。

    2. 总体方案

    本项目不采用“自动运行的 Team Agent 系统”，而采用更适合课程作业的轻量方案：

    三位组员各自承担明确角色，并使用自己的 AI 工具辅助完成本角色工作；所有文档、代码、日志和进度统一沉淀到同一个 Git 仓库中。

    这套方案的本质是：

    • 人负责最终判断和交付。
    • AI 负责辅助分析、生成初稿、拆任务、检查问题。
    • Git 仓库负责协作、留痕和版本管理。
    • 文档负责在组员之间交接信息。

    3. 角色分工

    本项目默认按 3 人分工，分别承担以下角色。

    3.1 组员 A：项目总控与文档负责人

    角色定位：

    • Lead / PM / 协作总控

    主要职责：

    • 统一维护项目目标、范围和优先级。
    • 组织分工和阶段推进。
    • 维护需求文档、任务清单、周报和汇报材料。
    • 检查其他两位组员的文档是否齐全。
    • 负责向老师或验收环节统一讲解项目整体方案。

    必须交付的内容：

    • docs/prd.md
    • docs/tasks.md
    • docs/weekly-report-weekX.md
    • docs/progress.md
    • demo/presentation-outline.md

    日常工作重点：

    • 每周更新任务状态。
    • 每次开会后同步纪要。
    • 每次里程碑前检查文档是否缺失。

    不负责：

    • 不主导核心算法实现。
    • 不直接承包全部代码。

    3.2 组员 B：数据结构与算法负责人

    角色定位：

    • Data & Algorithm

    主要职责：

    • 负责项目中所有核心数据结构设计。
    • 负责道路图、节点、边、权重等图模型设计。
    • 负责推荐、查找、路径规划等核心算法方案。
    • 输出算法复杂度分析和方案对比。
    • 向实现负责人说明算法接口需求。

    必须交付的内容：

    • docs/schema.md
    • docs/graph-model.md
    • docs/algorithm-design.md
    • docs/test-plan.md 中的算法测试部分

    日常工作重点：

    • 优先完成可落地的主方案。
    • 对关键算法做至少一轮样例验证。
    • 确保算法方案和课程题目要求一致。

    不负责：

    • 不独自决定系统技术栈。
    • 不擅自修改产品范围。

    3.3 组员 C：系统实现与集成负责人

    角色定位：

    • System & Implementation

    主要职责：

    • 负责技术选型、代码工程搭建和主要功能实现。
    • 负责数据库落地、接口设计和代码联调。
    • 负责前端或演示界面最小可用实现。
    • 负责系统运行、调试和最终可演示版本。
    • 负责打包运行说明。

    必须交付的内容：

    • docs/architecture.md
    • docs/api.md
    • src/ 下的主要代码
    • README.md 中的运行说明
    • demo/ 下的演示截图或演示脚本

    日常工作重点：

    • 先实现最小可运行闭环。
    • 根据文档做模块化开发，不随意改需求。
    • 合并前确保代码可运行。

    不负责：

    • 不单独变更核心算法方案。
    • 不跳过文档直接大规模改代码。

    4. 每位组员如何使用 AI

    每位组员都可以使用自己的 AI 工具，但只允许围绕自己的职责使用，避免角色混乱。

    4.1 组员 A 的 AI 使用方式

    适合让 AI 协助：

    • 梳理需求
    • 拆分任务
    • 生成周报初稿
    • 整理汇报提纲
    • 检查文档完整性

    不适合直接交给 AI：

    • 最终分工决策
    • 项目范围取舍
    • 验收策略判断

    4.2 组员 B 的 AI 使用方式

    适合让 AI 协助：

    • 对比数据结构方案
    • 推导算法思路
    • 生成复杂度分析初稿
    • 帮助整理算法文档

    不适合直接交给 AI：

    • 不经验证直接采用算法
    • 不结合题目数据规模就确认方案

    4.3 组员 C 的 AI 使用方式

    适合让 AI 协助：

    • 生成代码草稿
    • 搭建项目骨架
    • 编写接口样例
    • 解释报错
    • 补充测试样例

    不适合直接交给 AI：

    • 一次性生成整个项目并直接提交
    • 未阅读代码就合并进主分支

    5. 仓库作为协作中心

    整个项目的协作中心是同一个 Git 仓库。

    仓库承担以下作用：

    • 保存代码
    • 保存文档
    • 保存工作日志
    • 保存阶段进度
    • 保存演示和答辩材料
    • 记录成员协作过程

    原则上，所有正式内容都必须进入仓库，避免散落在聊天记录、个人电脑文件夹或临时文档里。

    7. 关键文件说明

    为了让每个人一看就知道该写什么，下面说明关键文件的用途和责任人。

    7.1 README.md

    用途：

    • 项目简介
    • 小组成员与分工
    • 技术栈说明
    • 运行方式
    • 仓库结构说明

    责任人：

    • 组员 C 主写
    • 组员 A 审核补充分工信息

    7.2 docs/prd.md

    用途：

    • 写清楚系统做什么
    • 哪些功能是必须做
    • 哪些功能是增强项
    • 用户使用流程

    责任人：

    • 组员 A

    7.3 docs/tasks.md

    用途：

    • 列出当前所有任务
    • 标明负责人、优先级、状态、截止时间

    责任人：

    • 组员 A

    7.4 docs/progress.md

    用途：

    • 用于跟踪项目整体阶段进度
    • 记录里程碑是否完成

    责任人：

    • 组员 A

    7.5 docs/schema.md

    用途：

    • 写清楚核心实体、字段、关系和数据字典

    责任人：

    • 组员 B

    7.6 docs/graph-model.md

    用途：

    • 写清楚道路图模型
    • 节点、边、权重、拥挤度、交通工具规则

    责任人：

    • 组员 B

    7.7 docs/algorithm-design.md

    用途：

    • 推荐算法
    • 查找算法
    • 路径规划算法
    • 算法复杂度分析
    • 方案选择理由

    责任人：

    • 组员 B

    7.8 docs/architecture.md

    用途：

    • 技术栈
    • 模块划分
    • 系统结构
    • 数据流

    责任人：

    • 组员 C

    7.9 docs/api.md

    用途：

    • 接口路径
    • 请求参数
    • 返回结构
    • 错误码

    责任人：

    • 组员 C

    7.10 docs/ui-spec.md

    用途：

    • 页面列表
    • 页面跳转
    • 关键操作流程
    • 地图和路径展示说明

    责任人：

    • 组员 C 主写
    • 组员 A 可补充演示逻辑

    7.11 docs/test-plan.md

    用途：

    • 算法测试
    • 功能测试
    • 边界测试
    • 验收测试

    责任人：

    • 组员 B 负责算法部分
    • 组员 C 负责系统功能部分
    • 组员 A 负责验收清单部分

    7.12 logs/

    用途：

    • 记录每位成员的工作内容
    • 记录使用了什么 AI 工具
    • 记录做了什么决策

    责任人：

    • 各自维护自己的日志文件

    8. 分支与合并规则

    为了避免多人互相覆盖内容，建议使用固定分支策略。

    8.1 分支命名

    建议至少保留以下分支：

    • main
    • feature/lead
    • feature/algorithm
    • feature/system

    含义：

    • main：稳定、可运行、可展示版本
    • feature/lead：组员 A 的文档与协作分支
    • feature/algorithm：组员 B 的算法与文档分支
    • feature/system：组员 C 的实现与接口分支

    8.2 合并规则

    • 不直接在 main 上随意修改。
    • 每位组员优先在自己的分支上提交。
    • 合并到 main 前，至少通知另外两位组员。
    • 合并前确认不会覆盖别人的最新修改。
    • 大改动先更新文档，再合并代码。

    8.3 提交信息规范

    建议使用统一格式：

    [角色] 动作: 简述内容

    示例：

    [Lead] update: 完成 prd 初稿和任务拆分
[Algorithm] add: 补充最短路径方案和复杂度分析
[System] fix: 修正接口返回结构并更新运行说明

    9. 日常协作规则

    为了让协作不失控，必须遵守以下规则。

    9.1 文档先行

    • 需求不清，不写代码。
    • 算法未定，不锁定实现方式。
    • 接口未定，不做大规模联调。
    • 页面未定，不做完整演示脚本。

    9.2 一人主责，其他人配合

    • 每份核心文档只有一位主负责人。
    • 其他人可以提建议，但不应绕过负责人直接重写。

    9.3 改动先同步

    • 凡是会影响他人工作的改动，先在群里同步。
    • 例如：改接口、改字段、改算法、改目录结构。

    9.4 问题及时留痕

    • 不在群里口头说完就结束。
    • 讨论后的结论要写入文档或日志。

    9.5 AI 产出必须人工确认

    • AI 给出的内容只能作为草稿。
    • 最终版本必须由负责人阅读、修改和确认后再提交。

    10. 每周工作节奏

    建议固定每周一个小循环，减少临时抱佛脚。

    每周一

    • 组员 A 更新本周任务。
    • 确认本周目标和截止时间。

    每周中段

    • 各自推进自己的模块。
    • 发现阻塞及时同步。

    每周末

    • 汇总本周完成情况。
    • 更新周报。
    • 更新日志。
    • 确认下周计划。

    11. 推荐的项目阶段计划

    下面是一套可直接执行的阶段安排，适合课程设计推进。

    阶段 1：启动与定范围

    目标：

    • 明确做哪些功能
    • 明确三人分工
    • 建好仓库结构

    完成标准：

    • 建立仓库
    • 建好 docs/、logs/、src/、demo/
    • 完成 docs/prd.md 初稿
    • 完成 docs/tasks.md 初稿

    责任重点：

    • 组员 A 主导

    阶段 2：核心数据与算法设计

    目标：

    • 确认数据结构和核心算法方案

    完成标准：

    • 完成 docs/schema.md
    • 完成 docs/graph-model.md
    • 完成 docs/algorithm-design.md
    • 确定推荐、查找、路径规划的主方案

    责任重点：

    • 组员 B 主导

    阶段 3：系统架构与接口设计

    目标：

    • 确定技术栈和实现边界

    完成标准：

    • 完成 docs/architecture.md
    • 完成 docs/api.md
    • 完成基础工程初始化

    责任重点：

    • 组员 C 主导

    阶段 4：最小可运行闭环实现

    目标：

    • 做出可演示的核心功能

    建议优先完成：

    13. 推荐的首次启动步骤

    3. 在 README.md 中写清楚三人分工。
    4. 由组员 A 完成 docs/prd.md 初稿。

    15. 风险与应对

    风险 1：文档和代码脱节

    表现：

    • 文档写的是一套，代码实现是另一套。

    应对：

    • 先更新文档，再改代码。
    • 合并前检查文档是否同步。

    风险 2：组员各做各的，最后拼不起来

    表现：

    • 每个人都很忙，但系统无法联调。

    应对：

    • 每周必须同步一次接口和依赖关系。
    • 任务拆分必须围绕同一个最小闭环。

    风险 3：AI 输出大量内容但质量不稳定

    表现：

    • 内容看起来很多，但经不起验收。

    应对：

    • AI 只做草稿。
    • 负责人必须二次审核并修改。
    • 关键算法必须手动验证样例。

    风险 4：后期文档补不齐

    表现：

    • 程序有了，但报告、周报、说明书缺失。

    应对：

    • 从第一周开始同步维护文档和日志。
    • 组员 A 每周检查一次缺漏。

    16. 对老师或答辩时的说法

    如果需要向老师解释你们为什么没有“部署真正的 Team Agent 系统”，建议这样表述：

    我们采用的是轻量化的 Agent Team 协作思路。由于课程作业更强调项目落地和算法实现，我们没有搭建复杂的自动化多 Agent 编排平台，而是采用“角色化分工 + 各自 AI 辅助 + 共享 Git 仓库 + 文档驱动协作”的方式实现。每位成员对应一个核心角色，并使用 AI 工具辅助完成该角色任务，所有产出统一进入仓库，以保证过程可追踪、协作清晰、成果可验收。

    这个说法合理、稳妥，而且符合课程讲义对“智能体使用能力”的要求。

    17. 最终执行标准

    只要你们做到下面这些，这套方案就算真正落地了：

    • 三人角色明确
    • 仓库结构完整
    • 文档持续更新
    • 代码围绕文档实现
    • 每周有周报和日志
    • 能讲清楚 AI 在各角色中的作用
    • 最终系统可运行、可演示、可说明

    18. 一句话总结

    这套方案的核心不是“搭一个复杂的 Team Agent 系统”，而是：

    让三位组员各自承担稳定角色，用各自的 AI 工具辅助工作，并通过共享 Git 仓库和标准文档完成真正可执行的协作。
